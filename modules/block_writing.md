# ブロック・ライティング
**ブロック・ライティング** とは知的生産手法の一つであり、単一の目的を持ったテキストの塊（ブロック）をつくっていくことで探索を前進させるものである。

## ブロック
以下を持つ:

- 1 つの名前（必須 or 不要 or 任意）
- 1 つのレター（ブロックの種類を示すアルファベット1文字または2文字）
- 内容（n行）

なお、ブロックはテキストのみから構成される。画像その他バイナリデータは含まない。

## ページ
ページとは複数のブロックを持つ単位である。

通常はファイルを指すはずだ。つまりブロック・ライティングは、テキストファイルとして運用するものである。リッチテキストなエディタやアプリでも実現できないことはないが、単一ページ内に何十何百というブロックを並べるため、軽さが必要だ。リッチテキストなツールではおそらく難しいだろう。

## ブロックの種類
- a: Action ブロック
    - 具体的な行動が出てきたら、このブロックに並べる
    - タスクリスト、TODO、アクションリストに相当する
    - ブロック名は任意
- b: Brast ブロック
    - Brast は Brain-Storming の略であり、ここでは特にひとりブレストを指す
    - 発散的な書き込みを行いたいときは、このブロックを使う
    - ブロック名は不要
- c: Concept ブロック
    - 概念として言語化するべきが出てきたら、このブロックに並べる
    - ブロック名は任意
- d: Direction ブロック
    - 戦略やビジョンや方針など、方向性に関する話はこのブロックに並べる
    - ブロック名は任意
- k: Kento ブロック
    - 何らかのテーマを一つ掲げ、それに関する検討内容を並べる
    - テーマを掲げたブレストブロックと同義となることが多い
    - ブロック名は必須
- cx: Context ブロック
    - 押さえておくべき文脈は、このブロックで整理しておく
    - ブロック名は不要
- s: Summary ブロック
    - ここまでのブロックを踏まえて、現時点の本質を示す
    - 本質とは「エグゼクティブ・サマリーのような本質的な見解」と「ネクストアクション」を指す
    - ブロック名は不要

なおブロック名については、必須のものは必ず指定しなければならない。任意のものは必要であれば書けばいい。不要のものは指定してはならない。これらの指示は意図的であり、いくつか例を挙げる:

- 検討ブロックでは「必須」となっているが、これは検討テーマを必ず指定せよ、ということである
- Summary ブロックでは「不要」とあるが、これはいたずらにブロック名をつけて抽象化するな、ということであり、本質として必要な情報は漏れなく並べるべきであるという思想を込めている
- 「任意」については、必要に応じて各自で好きな名前をつければ良い。つけなくても良い

## フロー
基本的な流れを解説する。

1: まずはテーマと文脈があるはずだ。創造的な仕事の場合はそれすらないかもしれないが、だとしても「とりあえず色々発散してみる」など仮テーマを掲げることはできる。これを Context ブロックにて文脈として言語化し、整理しておく。

2: 次にライティングを開始する。Brast や Kento などで発散を行い、行動が見えてきたら Action を、概念が見えてきたら Concept を、方向性に関する話は Directin を使って整理していく。たとえば Brast ブロックを 3 個くらいつくったところで、深堀りしたいトピックが 2 つ出たとしよう。この二つはそれぞれ Kento ブロックで検討すればよい。そうして発散や調査を繰り返すことで、自分が出しうる情報をとにかく出し切り、また整理していく。

3: そうして複数の――場合によっては数十以上、100 のブロックをつくっていくうちに、段々と本質が見えてくる。あるいは、少なくとも「現時点の自分が出せるものは出し切った」と言えるところまで辿り着ける。無論、現実的にはリソースの問題があり、本当はもっと出せるが二日しか費やせなかった、といったこともありえる。いずれにせよ、きりのいいところで Summary ブロックをつくる。

4: s ブロックの結果を踏まえて、もう一度 1: から回すか、それともブロック・ライティングを終えるかを判断する。回す場合は、Context ブロックとして Summary ブロックの内容を足す。つまり n 回目を開始するときの Context ブロックは、一番最初（1 回目のとき）にまとめた文脈と n-1 回目の Summary ブロックの内容を含むはずである。

## 例
以下は markdown の見出し記法を使って 1-大見出し 1-ブロック で書いたものであり、最新のブロックを下から上に積み上げたとする。

```

# cx
- 元々の文脈
    - ...
- 一周目のサマリー
    - ...

===

# s
...

# d 個人レベルでできそう
...

# d マネージャーに言えば何とかなるかも
...

# d 一個人ではどうにもならなそうなもの
...

# c
...

# k テーマ2
...

# k テーマ1
...

# b
...

# b
...

# b
...

# cx
...

```

一周目では 3 回のブレスト、2 回の検討を行い、方向性が比較的細かく整理されているのがわかる。すでに Summary までつくっており、これを踏まえて二周目に臨もうとしている。

なお、ここでは周の区切りとして `===` を使っているが、何でも良い。

## ページのサイズ感とエディタの習熟度
区切りではなくページを分けても良いが、ブロック・ライティングでは数千ブロック・数百ページ程度なら耐えられるはずである。文字数で言えば、100 万文字程度は耐えるはずだ。

もし 100 万、あるいは 10 万文字のページすらスムーズに扱えない場合は、おそらくツールに問題がある。すでに述べたとおり、2026 年 2 月現在でも、おそらくローカルのテキストエディタ一択となるだろう。

あるいは、ツールの習熟に問題があるかもしれない:

- ブロック間をショートカットキーで行き来する機能は必須であるが、IDE として名高い VSCode ではこの機能を持たない。強いて言えば「シンボル移動」の操作があるが、これでは不便だ。ショートカットキー一発で「一つ上」または「一つ下」の見出しを行き来できる必要がある。これくらいでないとシームレスにならない
