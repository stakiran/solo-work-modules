# JECTs メソッド
**JECTs メソッド** とは、個人の情報管理を統一的に行うフレームワークである。

## 6 観点
JECTs は以下の 6 観点から成る:

- 入口
    - 1: inject。外からの情報を受け付けて、個人領域（頭の中ではなく何でもいいので記録する）に移せ
- 個人領域
    - 2: object。未処理のもの。project、subject、reject、eject を確定する前の「とりあえず置いておくもの」や「まずは手元で検討してみるもの」
    - 3: project。実施するもの
    - 4: subject。日々の選択や行動を方向付けるもの
- 出口
    - 5: reject。捨てるもの。個人領域の外に出して、綺麗さっぱり忘れてしまえ。
    - 6: eject。外に提出するもの。特定少数から不特定多数まで、しかるべき相手に提出する。

## フロー
JECTs の流れは以下のように、4 段階から成る:

```
---> 1: inject ---> 2: object  ---> 3: reject
                       project
                       subject
                          |
                          V
                     4: eject

```

inject（外からの情報）はまず object に入れて処理する。

object からは、必要に応じて project や subject に移す。実施するものは project に移す。ここはタスク、TODO、プロジェクト、アクションなど呼び方が様々存在する。次に、直接実施するものではないが「健康に生きる」「英語を学習したい」のように行動を生み出す事柄となるものは subject である。これについても座右の銘、モットー、今週心がけたいことなど呼び方は無数にある。重要なのは行動を司る project と、行動を生み出す subject の二点を区別することである。なお、subject の考え方は、GTD で言えば高度 2000m 以上が該当し、PARA メソッドで言えば AoR(Area of Responsibility)が該当する。

個人領域で何でも扱っているとパンクしてしまうため、捨てる判断は意図的にしなければならない。これが reject である。object として残すのではない。残すことさえせず、捨ててしまうのである。

また社会で生きる以上、外への提出も求められる。誰に、いつ、何を提出するかはそれなりに練っておいた方が良い。このあたりを司るのが eject である。つまり適当に思いつきで過ごすのではなく、提出の戦略と仕組みを持つということだ。

## JECTs の価値
個人の情報管理、もっと言えば人生管理には無数の解がある。GTD のような古典とも言えるメソッドもあり、同じ GTD であっても人によって運用方法が違う。しかし、考えねばならない本質はシンプルである。JECTs はそのような、いわば人生管理の本質を 6 観点に凝縮したものである。このシンプルさとわかりやすさに価値がある。

ちなみに、このシンプルな運用さえも受け付けられないのであれば、あなたは自らを運用で管理するタイプではないのだろう。あるいは自己管理を行えるほどの能力や視座を持っていない。つまり自分個人の情報を管理する、もっと言えば人生を管理するという側面では「向いていない」ことになる。そういう意味で、JECTs は向き不向きを判別する道具としても役に立つ。
